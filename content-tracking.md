# 컨텐츠 구현 추적

기획서·테이블·코드를 교차 분석하여 컨텐츠 구현 현황을 추적하고, 업무 체크리스트를 관리합니다.

## 사용법

`/content-tracking <컨텐츠명> [기획서 참조] [옵션]`

- **컨텐츠명** (필수): `이벤트`, `미션`, `상점`, `슬롯` 등
- **기획서 참조** (선택): Confluence URL, 텍스트, 또는 `Design/` 폴더 내 파일 경로
- 예시: `/content-tracking 슬롯`
- 예시: `/content-tracking 이벤트 https://studio-g.atlassian.net/wiki/...`
- 예시: `/content-tracking 이벤트 update`

## 동작 분기

스킬은 **추적 문서 존재 여부**와 **기획서 제공 여부**에 따라 자동으로 동작을 결정합니다.

| 추적 문서 | 기획서 | 동작 |
|----------|--------|------|
| 없음 | 없음 | **신규 생성 (코드+테이블)**: devcomment 생성 → 테이블+코드 분석 → 현황 문서 + 체크리스트 생성 |
| 없음 | 있음 | **신규 생성 (풀 분석)**: 기획 분석 문서 생성 → devcomment 생성/읽기 → 테이블+코드 분석 → 현황 문서 + 체크리스트 생성 |
| 없음 | 기획서 생성 요청 | **자체기획문서 생성** (아래 "Phase 0" 참조) + devcomment 생성 |
| 있음 | 없음 | **갱신**: devcomment 읽기 → git diff 기반 증분 갱신. 이전에 기획 링크가 있었으면 재방문하여 변경 반영. 자체기획문서인 경우 기획문서 커밋 변경 기반 갱신 |
| 있음 | 있음 | **갱신 (기획 포함)**: devcomment 읽기 → 기획 재분석 + 코드/테이블 갱신 |

### 문서 위치
- 추적 문서: `.claude/tasks/tracking/tracking_{컨텐츠명}.md`
- 기획 분석 문서: `.claude/tasks/tracking/design_{컨텐츠명}.md`
- 개발자 코멘트: `.claude/tasks/tracking/devcomment_{컨텐츠명}.md`

## Phase 0: 자체기획문서 생성 (기획서 생성 요청 시)

기획서가 따로 제공되지 않고, 사용자가 기획서를 만들어달라고 요청한 경우 **자체기획문서**를 생성합니다.

### 0-1. 프로젝트 상태 판별

프로젝트에 해당 컨텐츠 관련 코드가 존재하는지 확인합니다.

- **코드 존재**: Service, Mediator, View, 테이블 등 관련 코드가 있는 경우 → **코드 분석 기반 기획문서 생성**
- **코드 없음 (빈 프로젝트)**: 관련 코드가 전혀 없는 경우 → **템플릿 기획문서 생성**

### 0-2a. 코드 분석 기반 기획문서 생성 (코드가 있는 경우)

1. Phase 2(테이블 분석) + Phase 3(코드 분석)을 먼저 수행하여 기존 구현 현황을 파악
2. 분석 결과를 기반으로 기획문서를 역생성: 코드/테이블에서 발견된 기능을 요구사항 형태로 정리
3. `.claude/tasks/tracking/design_{컨텐츠명}.md`에 아래 포맷으로 생성

```markdown
# 기획 분석: {컨텐츠명}

## 메타데이터
- **분석일**: {YYYY-MM-DD HH:MM}
- **기획서 출처**: 자체기획문서 (코드 분석 기반 역생성)
- **문서 유형**: 🔧 자체기획문서
- **분석 기반**: 기존 코드 및 테이블 구조 분석
- **분석 시점 Git Revision**: {commit hash}

## 요구사항 목록

### {카테고리 1}
| # | 요구사항 | 명확도 | 테이블 매핑 | 코드 참조 | 비고 |
|---|---------|--------|-----------|----------|------|
| 1 | {코드에서 추론한 요구사항} | 🟢 명확 | {테이블명.컬럼} | {파일:라인} | 코드에서 역추론 |

## 코드 기반 기능 목록
{분석한 코드에서 발견된 전체 기능을 구조화하여 정리}

## 추정 사항 / 확인 필요
{코드만으로는 의도를 확인할 수 없는 부분 목록}
```

### 0-2b. 템플릿 기획문서 생성 (빈 프로젝트인 경우)

해당 컨텐츠에 대한 일반적인 구조의 템플릿 기획문서를 생성합니다.

```markdown
# 기획 분석: {컨텐츠명}

- **분석일**: {YYYY-MM-DD HH:MM}
- **기획서 출처**: 자체기획문서 (템플릿)
- **문서 유형**: 🔧 자체기획문서
- **비고**: 기획 내용을 직접 채워주세요

## 요구사항 목록

### 기본 기능

|---|---------|--------|-----------|------|

- 각 카테고리의 요구사항을 구체화해주세요

- 테이블 매핑은 구현 시 자동으로 연결됩니다
```

### 0-3. 자체기획문서 갱신 규칙

자체기획문서는 외부 URL이 아닌 **문서 자체가 기획서**이므로, 갱신 시 다음 규칙을 따릅니다:

1. **기획문서 커밋 변경 감지**: `git log --follow -- .claude/tasks/tracking/design_{컨텐츠명}.md` 로 기획문서의 커밋 이력 확인
2. **추적문서의 마지막 갱신 Revision 이후** 기획문서에 변경이 있으면:
   - `git diff {마지막 revision}..HEAD -- .claude/tasks/tracking/design_{컨텐츠명}.md` 로 변경 내용 확인
   - 추가/수정/삭제된 요구사항을 추적 문서에 반영
3. **코드/테이블 변경**도 기존 갱신 로직(Phase 4 갱신 시)과 동일하게 처리
4. 변경 로그에 기획문서 변경과 코드 변경을 구분하여 기록:
   ```
   | {날짜} | {hash} | 자체기획문서 갱신: {요구사항 N개 추가/수정} |
   | {날짜} | {hash} | 코드 변경: {변경 요약} |
   ```

## Phase 1: 기획서 분석 (기획서 제공 시)

### 1-1. 기획서 수집

- **Confluence URL**: 브라우저 자동화(`@browser` MCP)로 **콘텐츠 순서 보존** 방식으로 읽기

  **Step 1: 콘텐츠 구조 분석** (`javascript_tool`)
  - `mcp__claude-in-chrome__tabs_context_mcp`로 탭 확인
  - `mcp__claude-in-chrome__navigate`로 URL 이동
  - `mcp__claude-in-chrome__javascript_tool`로 콘텐츠 컨테이너(`.ak-renderer-document` 등)를 찾고 구조 분석:
    - 콘텐츠 영역 내 이미지 개수 (UI 아이콘/이모지 제외, naturalWidth > 100 && naturalHeight > 100 필터)
    - 콘텐츠의 순서 맵 구축: `[{type:"text", content:"..."}, {type:"image", alt:"...", index:0}, ...]`

  **Step 2: 분기 - 이미지 유무에 따라 다른 방식**

  **A. 이미지 없음** → `get_page_text`로 텍스트만 수집
  - `mcp__claude-in-chrome__get_page_text`로 페이지 텍스트 콘텐츠 수집
  - 이 경우 추가 스크린샷 불필요

  **B. 이미지 있음** → 순서 보존 방식으로 텍스트+이미지 순차 수집
  - Step 1에서 만든 순서 맵을 따라 순차적으로 처리:
    - `type:"text"` → JS에서 추출한 텍스트 콘텐츠 사용
    - `type:"image"` → 해당 이미지로 `scroll_to` 후 `zoom`으로 영역 캡처 → 시각적 해석
  - **핵심**: 텍스트와 이미지의 원래 배치 순서를 유지하여, 이미지가 어떤 텍스트의 맥락에 속하는지 정확히 파악
  - 예시: `텍스트1 → 이미지1(플로우차트) → 텍스트2 → 이미지2(UI목업)` 순서 보존

  **콘텐츠 컨테이너 셀렉터 우선순위**:
  1. `[data-testid="renderer-content"]`
  2. `.ak-renderer-document`
  3. `#content-body`
  4. `#main-content`

- **텍스트**: 직접 사용
- **파일 경로**: Read 도구로 파일 읽기

### 1-2. 기획 분석 문서 생성

기획서 내용을 분석하여 `.claude/tasks/tracking/design_{컨텐츠명}.md`에 정리본을 생성합니다.

```markdown

## 메타데이터
- **분석일**: {YYYY-MM-DD HH:MM}
- **기획서 출처**: {URL 또는 파일경로}
- **분석 시점 기획서 버전**: {페이지 수정일 또는 파일 수정일이 확인 가능하면 기록}

## 요구사항 목록

### {카테고리 1}
| # | 요구사항 | 명확도 | 테이블 매핑 | 비고 |
|---|---------|--------|-----------|------|
| 1 | {요구사항 요약} | 🟢 명확 | {테이블명.컬럼} | |

| 3 | {요구사항 요약} | 🔴 불명확 | - | {불명확한 이유} |
```

### 1-3. 명확도 판정 기준

- **🟢 명확**: 테이블 컬럼/코드와 직접 매핑 가능. 구현 방법이 분명함
- **🟡 모호**: 방향은 있으나 세부사항 부족 (조건, 분기, 수치 등 미명시)
- **🔴 불명확**: 해석이 필요하거나 기술적으로 어떻게 구현할지 불분명

### 1.5-2. devcomment 읽기 (있을 때)

devcomment 파일이 이미 존재하면 내용을 읽어 이후 분석(Phase 2, 3, 4)에 **참고 컨텍스트**로 활용합니다.

- devcomment에 기록된 내용은 구현 현황 판정, 비고란 작성, 체크리스트 작성 시 부가 정보로 반영합니다.
- devcomment 내용이 기획서나 코드와 충돌하는 경우, 추적 문서의 비고란에 "devcomment와 차이 있음" 으로 표기합니다.

### 1.5-3. 사용자 구두 설명 캡처

사용자가 `/content-tracking` 요청 시 컨텐츠에 대해 구두로 설명하는 내용이 있으면:

1. 해당 내용을 devcomment 파일의 **"구두 전달 사항"** 섹션에 날짜와 함께 추가합니다.
2. 기존 구두 전달 사항은 삭제하지 않고 **누적 기록**합니다.

```markdown
## 구두 전달 사항

### {YYYY-MM-DD}
- {사용자가 전달한 내용 1}
- {사용자가 전달한 내용 2}

### {이전 날짜}
- {이전에 전달한 내용}
```

## Phase 2: 테이블 분석

1. **테이블 폴더 확인**: 프로젝트 상위의 `../table/excel/` 폴더 존재를 확인합니다.
   - 없으면 사용자에게 테이블 경로를 요청

2. **테이블 스캔**:
   - `../table/excel/` 폴더에서 컨텐츠 관련 Excel 파일 목록 스캔 (Glob 사용)
   - `Assets/Application/Scripts/Tables/Generated/`에서 대응하는 Generated 테이블 코드 분석
   - 테이블 스키마(컬럼명, 데이터 타입)에서 기능 항목 추출

3. **테이블 역매핑** (기획서 있을 때):
   - **기획서 → 테이블**: 각 요구사항에 대응하는 테이블 컬럼을 찾음
   - **테이블 → 기획서**: 테이블에 있지만 기획서에 언급되지 않은 항목을 찾음

## Phase 3: 코드 구현 현황 분석

1. **코드 검색**:
   - Service 코드: `Assets/00.Development/` 내 `*Service*.cs` 등
   - Mediator 코드: `Assets/00.Development/UIMediatorScripts/`
   - View 코드: `Assets/00.Development/UIViewScripts/`
   - 테이블 데이터를 실제로 참조/사용하는 코드를 Grep으로 탐색

2. **구현 상태 판정**:
   - **✅ 구현**: 테이블 데이터가 코드에서 참조되고 로직이 존재
   - **⚠️ 부분구현**: 일부 참조만 존재하거나 TODO/주석으로 남아있음
   - **❌ 미구현**: 테이블에 정의되어 있지만 코드에서 참조하지 않음

3. **Git Revision 기록**: `git rev-parse HEAD` 결과를 메타데이터에 기록

## Phase 4: 추적 문서 및 체크리스트 생성/갱신

### 신규 생성 시

`.claude/tasks/tracking/tracking_{컨텐츠명}.md`를 아래 포맷으로 생성합니다.

```markdown
# 컨텐츠 추적: {컨텐츠명}

## 메타데이터
- **생성일**: {YYYY-MM-DD HH:MM}
- **최종 갱신**: {YYYY-MM-DD HH:MM}
- **Git Revision**: {full commit hash}
```

### 갱신 시

1. **기획서 재확인**:

   **A. 외부기획서인 경우** (이전에 기획 링크가 있었던 경우):
   - 기존 `design_{컨텐츠명}.md`의 기획서 출처 URL을 재방문
   - 변경 내용이 있으면 기획 분석 문서를 갱신하고, 추적 문서에 반영

   **B. 자체기획문서인 경우** (문서 유형이 🔧 자체기획문서):
   - `git diff {기록된 revision}..HEAD -- .claude/tasks/tracking/design_{컨텐츠명}.md` 로 기획문서 변경 확인
   - 변경이 있으면 diff 내용을 분석하여 추가/수정/삭제된 요구사항을 추적 문서에 반영
   - 변경 로그에 `자체기획문서 갱신` 태그를 붙여 기록

2. **코드/테이블 변경 분석**:
   - `git diff {기록된 revision}..HEAD --name-only`로 변경 파일 목록 확인
   - 변경 파일 중 추적 대상만 필터링:
     - `Assets/Application/Scripts/Tables/Generated/` (테이블 코드)
     - `Assets/00.Development/` 내 Service, Mediator, View 파일
     - `../table/excel/` 내 Excel 파일
   - **커밋이 너무 많거나 변경 범위가 넓으면**: git diff 대신 해당 컨텐츠 관련 코드/테이블을 직접 재분석
   - 관련 변경이 없으면 "갱신할 내용 없음"을 보고하고 종료

3. **상태 갱신**:
   - ❌ → ✅: 미구현 기능에 코드가 새로 추가됨
   - ❌ → ⚠️: 부분적으로 코드가 추가됨
   - ⚠️ → ✅: 나머지 구현이 완료됨
   - 코드 참조(파일:라인) 최신화

4. **신규 항목 추가**: 새로 추가된 테이블 컬럼이나 기능 추가

5. **제거된 항목 처리**: 테이블에서 삭제된 항목 → `🗑️ 제거됨` 표시

6. **체크리스트 갱신**: 구현 완료된 항목 체크, 신규 미구현 항목 추가

7. **메타데이터 갱신**: Git Revision·최종 갱신일 업데이트

## 중요 규칙

- **테이블 컬럼과 코드 매핑은 보수적으로 판정합니다.** 확실하지 않으면 "⚠️ 부분구현" 또는 "❌ 미구현"으로 표시하고 비고에 이유를 적습니다.
- **코드 참조**는 `파일경로:라인번호` 형식으로 기록하여 빠르게 찾아갈 수 있게 합니다.
- **기획서 없이 생성한 경우**, 메타데이터에 "테이블+코드 기반"임을 명시합니다.
- Excel 파일을 직접 읽을 수 없는 경우, Generated 테이블 코드를 기준으로 분석합니다.
- 기존에 같은 컨텐츠의 추적 문서가 있으면 갱신 모드로 동작합니다.
- 테이블 역매핑에서 발견된 "기획서에 없는 테이블 기능"은 반드시 체크리스트에 기획자 확인 항목으로 추가합니다. 임의로 불필요하다고 판단하지 않습니다.
- 갱신 시 변경 로그는 **누적 기록**입니다. 이전 로그를 삭제하지 않습니다.
- **자체기획문서**는 사용자가 직접 기획서를 수정하며 진행하는 문서입니다. 외부 URL을 재방문하는 것이 아니라, 기획문서 파일 자체의 **git 커밋 변경 이력**을 기반으로 추적합니다.
- 자체기획문서의 경우 메타데이터에 반드시 `문서 유형: 🔧 자체기획문서`를 명시합니다.
- 각 Phase 완료 후 사용자에게 중간 보고하고 확인을 받습니다.
- 체크리스트의 "기획 확인 필요" 항목은 기획자에게 바로 전달할 수 있는 수준으로 작성합니다. 기술 용어보다 기획 관점의 질문을 우선합니다.
- **devcomment 파일은 모든 신규 생성 시 디폴트로 생성합니다.** 사용자가 구두 설명을 하지 않아도 빈 템플릿을 생성하여 나중에 기록할 수 있게 합니다.
- **devcomment의 내용은 부가 정보입니다.** 기획서나 코드 분석 결과와 동일한 가중치로 취급하지 않으며, 참고 컨텍스트로만 활용합니다.
- **devcomment의 구두 전달 사항은 누적 기록합니다.** 이전 기록을 삭제하지 않고 날짜별로 쌓아갑니다.
- **전체 워크플로우 순서**: design 문서 제작/읽기 → devcomment 생성/읽기 → 코드·테이블 분석 → tracking 문서 제작/갱신
