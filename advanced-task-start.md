# 고급업무시작

업무를 체계적으로 분석하고 문서화한 뒤 수행하는 스킬입니다.
세션이 끊겨도 문서만 있으면 바로 재개할 수 있도록 자립형 문서를 생성합니다.

## 사용법

`/advanced-task-start <업무 내용>`

## 수행 절차

사용자로부터 업무를 요청받으면, 아래 절차를 **반드시 순서대로** 수행하세요.

### Phase 1: 업무 분석 및 문서 작성

1. **업무 분석**: 요청된 업무를 깊이 분석합니다.
   - 코드베이스 탐색 (Glob, Grep, Read 등 활용)
   - 관련 파일, 아키텍처, 의존성 파악
   - 변경 범위(blast radius) 분석
   - 필요시 사용자에게 명확화 질문
   - **스크린샷/이미지 처리**: 사용자가 이미지(스크린샷, UI 디자인, 참조 이미지 등)를 제공한 경우:
     a. `.claude/tasks/screenshots/{YYYYMMDD}_{간략한이름}/` 폴더를 생성합니다.
     b. 제공된 이미지 파일을 해당 폴더로 복사합니다.
        - 파일명 규칙: `{NN}_{설명라벨}.{확장자}` (예: `01_ui_design.png`)
        - NN은 01부터 순번, 설명라벨은 영문 snake_case
        - Bash의 `cp` 명령어로 복사: `cp "{원본경로}" ".claude/tasks/screenshots/{task_id}/{파일명}"`
     c. 각 이미지에 대해 텍스트 설명을 작성합니다 (이미지 내용, 관련 업무 컨텍스트).
     d. 이미지가 임시 경로(예: `/tmp/`, AppData 등)에 있는 경우, 반드시 복사 후 원본 경로가 사라져도 문제없는지 확인합니다.

2. **업무계획서 작성**: `.claude/tasks/` 폴더에 다음 파일을 생성합니다.

   **파일명**: `task_{YYYYMMDD}_{간략한이름}.md`

   **내용 구조**:
   ```markdown
   # 업무계획서: {업무 제목}

   - 작성일: {날짜}
   - 상태: 진행중

   ## 1. 업무 요약
   {사용자 요청 원문 및 요약}

   ## 2. 분석 결과
   ### 관련 파일
   {파일 경로 목록 + 각 파일의 역할 설명}

   ### 아키텍처 컨텍스트
   {현재 코드 구조, 패턴, 의존성 분석 결과}

   ### 핵심 코드 스니펫
   {변경 대상이 되는 주요 코드 블록 - 재개 시 맥락 파악용}

   ## 3. 구현 계획
   ### 접근 방식
   {선택한 구현 전략과 이유}

   ### 변경 사항 목록
   {파일별 변경 내용 상세 기술}

   ### 주의 사항
   {사이드이펙트, 리스크, 주의할 점}

   ## 4. 검증 방법
   {완료 후 어떻게 검증할 것인지 - Phase 4에서 자동/수동 검증 수행}

   ### 예상되는 사용자 확인 항목
   {구현 계획 기반으로 사람이 직접 확인해야 할 항목을 미리 나열}
   - {UI 변경이 있으면: 화면 레이아웃 확인}
   - {이펙트가 있으면: Play 모드에서 연출 확인}
   - {서버 통신이 있으면: 개발 서버 연결 테스트}
   - 등

   ## 5. 참조 이미지
   {사용자가 제공한 이미지가 없으면 "해당 없음"으로 표시}

   이 섹션의 이미지는 세션 재개 시 Read 도구로 열어볼 수 있습니다.

   | # | 파일 경로 | 설명 | 관련 섹션 |
   |---|----------|------|----------|
   | 1 | .claude/tasks/screenshots/{task_id}/{파일명} | {이미지 내용 설명} | {관련 섹션 번호} |
   ```

3. **TodoList 작성**: 같은 폴더에 체크리스트 파일을 생성합니다.

   **파일명**: `todo_{YYYYMMDD}_{간략한이름}.md`

   **내용 구조**:
   ```markdown
   # TodoList: {업무 제목}

   업무계획서: task_{YYYYMMDD}_{간략한이름}.md
   작성일: {날짜}
   최종수정: {날짜}

   ## 진행 상황

   - [ ] Todo 1: {구체적인 작업 내용}
   - [ ] Todo 2: {구체적인 작업 내용}
   - [ ] Todo 3: {구체적인 작업 내용}
   ...

   ## 완료 기록
   {완료된 항목에 대한 간단한 메모 - 변경된 파일, 수정 내용 등}

   ## 검증 체크리스트
   {Phase 4 완료 시 자동 생성됨}

   ### AI 자동 검증 결과
   {Phase 4-1 결과가 여기에 기록됨}

   ### 사용자 확인 체크리스트
   {Phase 4-2에서 생성된 항목이 여기에 기록됨}
   ```

### Phase 2: 사용자 확인

- 작성된 업무계획서와 TodoList를 사용자에게 요약해서 보여줍니다.
- 사용자 승인을 받은 후 Phase 3으로 진행합니다.

### Phase 3: 업무 수행

> **⚠️ 필수 확인**: Phase 3 진행 중 사용자가 요구사항을 변경/추가/삭제하면, **코드 수정 전에 반드시 "중간 요구사항 변경 처리" 절차를 먼저 수행**하세요. 문서 업데이트 없이 코드만 수정하는 것은 금지됩니다. → 아래 "중간 요구사항 변경 처리 (필수)" 섹션 참조

1. **TodoWrite 도구**로 진행 상황을 실시간 추적합니다.
2. Todo 항목을 **하나씩 순서대로** 수행합니다.
3. 각 Todo 완료 시 **스크립트로 체크** (파일 직접 읽기/쓰기 금지):
   ```bash
   bash ~/.claude/scripts/todo-check.sh <todo_파일경로> <미체크항목번호> [완료메모]
   ```
   - `<미체크항목번호>`: 미체크 항목 중 N번째 (1이면 첫 번째 남은 항목)
   - 예: `bash ~/.claude/scripts/todo-check.sh .claude/tasks/todo_20260305_feature.md 1 "파일A, 파일B 수정"`
   - 스크립트가 자동으로: 체크 표시, 완료 기록 추가, 최종수정일 갱신
   - TodoWrite 도구에서도 completed로 마킹
   - 진행 상태 확인: `bash ~/.claude/scripts/todo-check.sh <todo_파일경로> --status`
4. 모든 Todo 완료 시 → **Phase 4로 진행**

### Phase 4: 완료 검증

모든 구현 작업이 끝나면, 아래 절차로 결과를 검증합니다.

#### 4-1. AI 자동 검증

다음 항목들을 AI가 자동으로 수행하고 결과를 보고합니다:

1. **계획 대비 변경 대조**:
   - `git diff`로 실제 변경된 파일 목록 확인
   - 업무계획서의 "변경 사항 목록"과 대조
   - 누락된 변경이나 계획에 없는 변경이 있으면 보고

2. **코드 품질 검사**:
   - TODO/FIXME/HACK 주석이 남아있는지 확인
   - 구독 누락 검사: UniRx `.Subscribe()`에 `.AddTo()` 누락 여부
   - null 참조 위험이 명백한 코드가 없는지 확인

3. **TodoList 완전성 확인**:
   - `bash ~/.claude/scripts/todo-check.sh <todo_파일경로> --status` 로 확인
   - 미완료 항목이 있으면 이유를 정리

#### 4-2. 사용자 확인 체크리스트 생성

변경된 파일과 작업 내용을 분석하여, **사람이 직접 확인해야 하는 항목**을 체크리스트로 생성합니다.

다음 기준으로 자동 분류합니다:

| 변경 유형 | 확인 항목 예시 |
|----------|--------------|
| UI 프리팹 (Panel_*, Popup_*, Piece_*) | "Unity에서 해당 화면을 열고 레이아웃/정렬/텍스트가 정상인지 확인" |
| 이펙트/파티클 (fx_*, Effect/) | "Play 모드에서 이펙트 재생 확인 (타이밍, 위치, 색상)" |
| Spine 애니메이션 | "Play 모드에서 스파인 연출이 정상 재생되는지 확인" |
| DOTween/VisualAction 연출 | "Play 모드에서 트윈 연출이 의도대로 동작하는지 확인" |
| 사운드/진동 | "실기기에서 사운드/햅틱 재생 확인" |
| 스크롤/리스트 UI | "아이템 수가 많을 때 스크롤이 정상 동작하는지 확인" |
| 해상도/레이아웃 | "다양한 해상도에서 UI가 깨지지 않는지 확인" |
| 서버 통신 (Protocol/API) | "개발 서버 연결 상태에서 정상 동작 확인" |
| 로컬 데이터 저장 | "앱 재시작 후 데이터가 유지되는지 확인" |
| 테이블 데이터 참조 | "테이블 값 변경 시 UI/로직에 정상 반영되는지 확인" |
| 플랫폼 분기 (#if UNITY_IOS 등) | "iOS/Android 각각에서 동작 확인" |

**체크리스트 형식** (업무계획서와 TodoList에 추가):

```markdown
## 6. 완료 검증

### AI 자동 검증 결과
- ✅ 계획 대비 변경: 일치
- ✅ 코드 품질: 이상 없음
- ✅ TodoList: 전체 완료

### 사용자 확인 체크리스트
> 아래 항목은 Unity Editor 또는 실기기에서 직접 확인이 필요합니다.

- [ ] {확인 항목 1}: {구체적으로 무엇을 어디서 확인해야 하는지}
- [ ] {확인 항목 2}: {구체적으로 무엇을 어디서 확인해야 하는지}
...
```

#### 4-3. 최종 보고

1. AI 자동 검증 결과를 요약하여 보고합니다.
2. 사용자 확인 체크리스트를 표시합니다.
3. 업무계획서에 "6. 완료 검증" 섹션을 추가합니다.
4. TodoList에 "검증 체크리스트" 섹션을 추가합니다.
5. 업무계획서의 상태를 **"검증 대기"**로 변경합니다.
   - 사용자가 체크리스트를 모두 확인한 후 "완료"로 최종 변경합니다.
   - 사용자가 확인 중 문제를 발견하면 수정 작업을 추가로 진행합니다.

## 중간 요구사항 변경 처리 (필수)

> **⚠️ 이 절차는 Phase 3, Phase 4 진행 중 사용자가 요구사항을 변경/추가/삭제할 때 반드시 따라야 합니다.**
> **절대 코드 수정을 먼저 하고 문서를 나중에 업데이트하지 마세요. 문서 → 코드 순서를 반드시 지키세요.**
> **이 절차를 건너뛰면 세션 재개 시 문서와 실제 코드가 불일치하여 작업이 꼬입니다.**

사용자가 업무 수행 도중 **요구사항 변경, 추가, 삭제, 버그 수정 요청, 피드백**을 제공하면, **코드를 수정하기 전에** 다음을 **반드시** 수행합니다:

### 순서 1: 업무계획서(`task_*.md`) 즉시 업데이트
- "업무 요약" 섹션에 변경된 요구사항 반영
- "구현 계획" 섹션의 접근 방식, 변경 사항 목록 수정
- 변경 이력을 문서 하단에 기록 (예: `### 변경 이력` → `- {날짜}: {변경 내용 요약}`)
- 변경으로 인해 새로운 관련 파일이나 주의사항이 생기면 해당 섹션도 업데이트
- 추가 이미지가 제공되면 "참조 이미지" 섹션에 추가 (기존 순번 이어서)

### 순서 2: TodoList(`todo_*.md`) 즉시 업데이트
- 불필요해진 Todo 항목 삭제 또는 취소 표시 (`~취소됨~`)
- 새로 필요한 Todo 항목 추가 (변경요청임을 명시: `(변경요청) ...`)
- 기존 항목의 내용이 변경되면 수정
- "최종수정" 날짜 갱신

### 순서 3: TodoWrite 도구 동기화
- TodoList 파일 변경에 맞춰 TodoWrite 도구의 항목도 함께 업데이트

### 순서 4: 사용자 확인
- 변경된 계획과 Todo를 사용자에게 요약 보고
- 승인 후 변경된 계획에 따라 업무 계속 수행

### 순서 5: 코드 수정 수행
- 위 1~4 단계가 완료된 후에만 실제 코드 변경 작업을 시작합니다.

> **핵심 원칙**: 문서는 항상 **현재 업무의 최신 상태**를 반영해야 합니다. 요구사항이 변경되었는데 문서가 이전 상태로 남아있으면, 세션 재개 시 혼란을 초래합니다.
>
> **위반 시 체크**: 코드를 수정한 후 `task_*.md`와 `todo_*.md`가 업데이트되지 않았다면, 이는 절차 위반입니다. 즉시 문서를 업데이트하세요.

## 중요 규칙

- **요구사항 변경 시 문서 먼저 업데이트**: 사용자가 피드백, 변경, 추가 요청을 하면 **코드 수정 전에** 반드시 `task_*.md`와 `todo_*.md`를 먼저 업데이트합니다. 이것은 가장 중요한 규칙입니다.
- 문서는 **세션 독립적**이어야 합니다. 새 세션에서 문서만 읽고도 맥락을 100% 파악할 수 있어야 합니다.
- 코드 스니펫은 변경 전/후를 명확히 구분합니다.
- 파일 경로는 항상 절대경로 또는 프로젝트 루트 기준 상대경로로 기록합니다.
- Todo 항목은 하나의 명확한 작업 단위로 쪼갭니다 (너무 크지도, 너무 작지도 않게).
- 스크린샷/이미지가 제공된 경우, **텍스트 설명**을 충분히 상세하게 작성합니다. 이미지를 볼 수 없는 상황에서도 설명만으로 맥락을 파악할 수 있어야 합니다.
- 스크린샷 경로는 프로젝트 루트 기준 상대경로(`.claude/tasks/screenshots/...`)로 기록합니다.
